# Отчет по Практической работе №7
**Тема:** Рефакторинг проекта. Переход к архитектуре MVC.

## 1. Цель работы
Глубокий рефакторинг разработанной информационной системы. Переход от процедурного стиля программирования к объектно-ориентированному (ООП) с применением архитектурного паттерна **MVC (Model-View-Controller)**. Разделение бизнес-логики, механизмов доступа к данным и пользовательского интерфейса.

## 2. Архитектура проекта (MVC)
Приложение перестроено согласно принципам "Чистой архитектуры" и паттерна MVC:

### Core (Ядро)
*   **`Router.php`:** Единая точка входа. Анализирует GET-параметр `action` и делегирует управление соответствующему контроллеру и его методу.
*   **`Auth.php`:** Выделенный класс для обработки механизмов авторизации и проверки прав доступа.

### Models (Модели) — Работа с данными
*   **`Database.php`:** Реализация паттерна **Singleton** для управления подключением к MySQL. Гарантирует наличие только одного активного соединения с БД.
*   **`Statistic.php`, `Dish.php`, `User.php`:** Классы-сущности, инкапсулирующие логику работы с конкретными таблицами БД (CRUD операции).

### Controllers (Контроллеры) — Бизнес-логика
*   **`BaseController.php`:** Базовый класс, содержащий общую логику для всех контроллеров (инициализация персонализации, тем оформления, языковых настроек и метод `render` для вывода представлений).
*   **`HomeController`, `MenuController`, `AdminController` и др.:** Наследуются от базового контроллера, обрабатывают конкретные действия пользователя и готовят данные для отображения.

### Views (Представления) — Интерфейс
*   Директория `views/` содержит чистые PHP/HTML шаблоны. Представления не знают о логике приложения и БД, они лишь отображают переменные, переданные им контроллером через метод `extract()`.

## 3. Основные изменения при рефакторинге
1.  **Автозагрузка (PSR-4):** Использование Composer для автоматического подключения классов по их пространствам имен (`App\Controllers`, `App\Models`). Больше нет нужды в ручном `require_once` для каждого файла.
2.  **Инкапсуляция:** Вся логика работы с БД вынесена из файлов отображения в методы моделей.
3.  **Маршрутизация:** Процедурные файлы (типа `dynamic1.php`, `upload.php`) заменены на единый `index.php`, который через `Router` определяет, какой код исполнить.
4.  **Повторное использование кода:** Общие задачи (проверка куки, выбор темы, рендеринг) реализованы один раз в `BaseController`.

## 4. Схема работы обновленной системы
1. Запрос пользователя (`index.php?action=menu`).
2. `Router` создает экземпляр `MenuController` и вызывает метод `index()`.
3. `MenuController` обращается к модели `Dish::getAll()`.
4. Модель `Dish` через `Database::getInstance()` запрашивает данные из MySQL.
5. Контроллер получает данные и передает их в `render('menu', ['dishes' => ...])`.
6. Представление `views/menu.php` генерирует HTML и отправляет его пользователю.

## 5. Вывод
Рефакторинг позволил превратить разрозненные скрипты в структурированное приложение. Применение MVC и ООП значительно повысило читаемость кода, упростило его тестирование и дальнейшее расширение (например, добавление новых страниц теперь требует создания лишь одного метода в контроллере и одного файла представления).
